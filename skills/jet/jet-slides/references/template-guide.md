## How to Use This Guide

When creating a presentation from the template, you will:
1. Clone the template using `slides-cli create from-template`
2. Keep slides you need and delete the rest using `slides-cli edit delete-slides`
3. Duplicate slides to create additional pages of the same type using `slides-cli edit duplicate-slide`
4. Replace placeholder text using `slides-cli edit replace-text` or `slides-cli edit set-text`
5. Use `slides-cli edit batch` for complex operations

### Common Placeholder Patterns
- **Footer**: `|  Month Year  |  Title of project` — replace with actual date and project name
- **Project tags**: `[Project tags]` — replace with value stream tags
- **Body text**: `Lorem ipsum dolor sit amet...` — replace with actual content
- **Names**: `Person's name` or `Name Surname` — replace with actual names
- **Cover line**: `Project name\nValue Stream` — replace with project name and stream
- **Cover author**: `Name Surname | Month Year` — replace with author info

## Typical Presentation Structure

A well-formed JET presentation follows this pattern:

1. **Cover page** (1 of slides 0–12) — pick the cover whose colour/image best fits the topic
2. **Contents page** (slide 36) — optional, for longer decks
3. **Divider** (slide 14 or 35) — section break
4. **Project context** (slides 37–39) — background, objectives
5. **Executive summary** (slide 40) — key takeaways
6. **Content slides** — mix of:
   - Three points / insights (50–52)
   - Quotes (43–44)
   - Graphs / data (67–72)
   - Card pages (105–113)
   - Split pages (99, 126–139)
   - Column grids (86–98)
   - Call-out text (59–62, 82–84)
7. **Dividers** between sections
8. **Conclusion** (slide 150) or **Summary** (slide 149)
9. **Thanks** (slide 85) or **Thank you** (slide 176)

**Colour usage:** Use JET Orange dividers (slide 14) between sections to ensure orange appears throughout the deck. Cover and closing slide colours are baked into the template and vary by cover option — this is by design.

## Visual Review Lessons Learned

These patterns were discovered by populating and reviewing all 58 unique slide types in the template. They apply to every deck built with this skill.

### 1. Character limits are the most critical constraint

Many visual issues stem from text exceeding the box capacity. Venn diagram circles (slide 48 area), narrow card columns, and Y-axis labels are the worst offenders. **Always check the element capacity tables in the slide catalogue sections above and § "Minimum Content Volume Per Slide Type" below before writing content for any element.**

### 2. Empty space is the #1 visual problem

Most content slides have 30-60% unused vertical space when populated with minimal content. The template was designed for presentations with substantial written content. **Never use a slide type without providing the minimum content volume** (see the "Minimum Content Volume Per Slide Type" section below).

### 3. Template images cannot be changed via text commands

Food/lifestyle images are baked into the template. Cover slides, intro slides, and card layouts all have fixed imagery. **Always check the Template Image Guide below to ensure the stock photo matches the presentation topic.** Use `insert-image` to replace images when possible.

### 4. Chart/graph areas and device mockups are not directly editable
Charts are decorative shapes or empty rectangles — use text-based numbers or `insert-image`. Device mockups (60, 61, 162-165) need `insert-image` for screenshots. See `slides-api-gotchas.md` § "Charts and graphs cannot be populated via API".

### 6. Group elements with placeholder text remain if not targeted

Group elements contain nested text (e.g., "Heading", "Lorem ipsum", "Person's name") that persists unless explicitly replaced with `replaceAllText`. **Always scan group children for placeholder text and replace all of it.**

### 7. Data consistency errors destroy credibility

Numbers must add up. If two data points show "23%" but a change column says "+4.2pp", the arithmetic is wrong. **Always verify that numeric data is internally consistent across all elements on a slide.**

### 8. Decorative shapes are fixed
Arrow indicators, severity badges, icons, and bubble shapes cannot be changed. Content must match the template's fixed decorative elements. See `slides-api-gotchas.md` § "Decorative shape elements cannot be modified".

### 9. Formatting restoration is mandatory after every set-text

This is the #1 systemic issue. Every `set-text` call strips ALL formatting — font family, weight, size, colour, and underline state. Without restoration, text falls back to Arial and the slide looks broken. **Batch all `updateTextStyle` restorations after each round of `set-text` calls.**

### 10. Sentence case violations are the most common brand error

Title Case headings appear professional but violate JET brand guidelines. **All text — headings, card titles, divider text, labels, footers, subtitles — must use sentence case.** Only capitalise the first word and proper nouns.

### 11. Text boxes are intentionally smaller than their background shapes
This is template padding by design, not a bug. Prefer shortening text over resizing boxes. See `slides-api-gotchas.md` § "Text Box Sizing: Intentional Padding by Template Design" for the full explanation and resize recipe.

### 12. Tabs slide (template 147) text boxes overflow with long content

The tabs slide has two editable body text boxes in the left column, each designed for ~17 characters of placeholder text. When populating with real content, keep text under 70 characters per cell. At 138+ characters, text crowds the bottom edge of the cell. Shorten to 2 short sentences max per cell.

## Vertical Centering Guide by Slide Type

After populating a deck, apply `contentAlignment: MIDDLE` to self-contained text elements with short text in tall containers. This dramatically improves aesthetics by eliminating top-heavy dead space. See `slides-api-gotchas.md` for the full recipe and warnings.

### Slide types that benefit from vertical centering

| Slide type | Elements to center | Why it helps |
|-----------|-------------------|--------------|
| Quote slides (43, 44) | Quote text boxes, attribution text | Short quotes in tall speech bubbles look unbalanced when top-aligned |
| Journey circles (63, 65, 66) | Circle/chevron labels | "Sign up", "Go live" etc. look better centred in their circle/chevron containers |
| Call-out short (82) | Main statement text | Single punchy statement on orange background — centering feels natural |
| Call-out medium (84) | Circle text element | Text inside the circular container |
| Card pages — number labels (75, 76) | Number shapes ("1", "2", etc.) | Large numbers in tall containers |
| Conclusion (150) | Body text block | Centres the statement in the large empty area |
| Company profile (146) | Section headings | Headings in tall header cells |
| Lows & Highs (55, 56) | Right-panel heading, bullet labels | Short heading text in tall header area |
| Graphs — stat numbers (67, 70) | Hero number displays | Single numbers like "34" or "6.3" in tall stat boxes |
| Competitive comparison (72) | Score cells | Single-char scores ("3", "A+") in tall narrow cells |
| Tabs (147) | Tab label elements | Tab labels in tall tab header cells |
| Introduction top 3 (148) | Number labels, point headings | Numbered points with short headings |
| Split with icons (141-144) | Icon label text | Short labels beside icon slots |
| Cover title (0-12) | Title text box (if self-contained) | Centres the two-line title vertically on the slide |
| Divider headline (14-27) | Headline text box | Centres the headline on the coloured background |
| Thanks / closing (85, 176) | Heading and contact elements | Stacked vertically, not parallel |
| Point page numbers (115-119) | "1", "2", "3" number labels | Self-contained decorative numbers |
| Stat number + label pairs (70) | Number and label boxes | Stacked vertically, not parallel |
| Callout boxes (59-62, 82-84) | Self-contained callout text | Single statement in a tall container |

### CRITICAL: Parallel/grid layouts must use TOP alignment

See `slides-api-gotchas.md` § "CRITICAL: Parallel/grid layouts must use TOP alignment" for the full explanation. Summary: MIDDLE centering in parallel layouts destroys horizontal baseline alignment.

| Slide type | Elements that MUST stay TOP-aligned |
|-----------|-------------------------------------|
| Monthly timeline (46) | Month boxes |
| Simple timeline (47) | Step description boxes |
| Timeline 5-phase (151) | Phase label+description boxes |
| 6-card grid (74, 75, 76) | Card body text (number labels can keep MIDDLE) |
| Journey map with severity (64) | Finding + severity descriptions |
| Breadcrumb sub-steps (77-80) | Sub-step descriptions |
| Process roadmap (152) | Description boxes below chevrons |
| Content grid cards (73-76) | Card body text with title+description |

### Slide types where vertical centering should NOT be applied

| Slide type | Elements to leave top-aligned |
|-----------|-------------------------------|
| NPS / card slides with label + body stacked | Label headings above body text (centering causes overlap) |
| Executive summary (40) | Body text area |
| 1-column grids (86-92) | Body text |
| Multi-column grids (94-98) | Body text in columns |
| Project context (37-39) | Section body text |
| Footers (all slides) | Footer text elements |
| Split pages (99, 126-139) | Body text on text side |
| Summary (149) | Body text blocks |
| Three points / insights (50, 51, 52) | Column body text |

### Rule of thumb

See `slides-api-gotchas.md` § "When to use `contentAlignment: MIDDLE`" for the full criteria. In brief: only apply MIDDLE to **short text** (<80 chars) in **tall self-contained containers** that are **not in parallel/grid layouts**. When in doubt, leave it top-aligned.
